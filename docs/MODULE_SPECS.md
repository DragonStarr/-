# Мини-ТЗ по 23 модулям

Общее правило для всех модулей: пользователь не получает инструкцию “как сделать руками”. Модуль собирает данные, готовит действие, показывает короткое превью “было -> станет” и ждет кнопку `ОК`, если действие влияет на деньги, цены, поставки, людей, права или спор. Каждое действие обязано иметь `skills`, `plugins`, минимум 10 `mcp_checks` и `answer_basis`, чтобы бот мог объяснить, на чем основана рекомендация.

## 1. Подключение кабинетов и единый каталог

1. Делает: подключает WB/Ozon/Яндекс Маркет по официальным токенам, проверяет права токена, строит единый каталог.
2. Источники ответа: официальные API площадок, ручной импорт тестового каталога, tenant-история синхронизаций; при отсутствии источников модуль показывает подключение данных, а не фиктивные товары.
3. БД: `accounts`, `products`, `stocks`, `orders`, `audit_log`; все записи с `tenant_id`.
4. Серверы: FastAPI принимает подключение, Celery синхронизирует каждые 15-60 минут, Redis держит лимиты.
5. ЛК/площадки: WB token scopes по доменам, Ozon `Client-Id` + `Api-Key`, ЯМ API key + campaign id.
6. Интерфейс бота: `Кабинеты -> Подключить -> Проверить`; при ошибке короткий текст и кнопка `Проверить еще раз`; при недоступном API владелец может загрузить свой каталог через backend endpoint.
7. Защита: токены шифруются, не логируются, в ответах показываются только последние 4 символа fingerprint.
8. Данные: чужие кабинеты и публичный скрейпинг ЛК запрещены; только свои данные пользователя.
9. Связи: питает все остальные модули товарами, остатками, заказами, отзывами, вопросами.
10. Эскалация: нет прав токена, истек ключ, лимит API или расхождение схемы API -> задача владельцу и запись в аудит.

## 2. Юнит-экономика, аналитика и аномалии

1. Делает: считает маржу SKU после комиссий, логистики, хранения, рекламы, возвратов, налогов и удержаний.
2. Источники ответа: заказы, финоперации, ставки площадок, пользовательские статьи затрат, правила тарифа.
3. БД: `orders`, `finance_ops`, `products`, витрины `pnl_daily`, `abc_segments`.
4. Серверы: ночной пересчет, пересчет по событию изменения тарифа, API для объяснения расчета.
5. ЛК/площадки: берет отчеты продаж/финансов только через официальные методы или импорт пользователя.
6. Интерфейс бота: `3 товара в минусе -> Показать расчет -> Поднять цену?`.
7. Защита: финансы видят `owner` и разрешенный `manager`; все правки затрат аудируются.
8. Данные: хранит точность расчета, дату источника и версию тарифной формулы.
9. Связи: дает сигналы в утренние дела, репрайс, прогноз, финмонитор.
10. Эскалация: если данных мало или тариф неизвестен, пишет “точность низкая” и не предлагает автоцену.

## 3. SEO-тексты карточек

1. Делает: готовит заголовок, описание и характеристики без фото/видео.
2. Источники ответа: карточка товара, поисковые запросы, правила категории, удачные прошлые тексты.
3. БД: `products`, `seo_drafts`, `seo_queries`, `feedback`.
4. Серверы: воркер проверяет неполные карточки раз в неделю, LLM генерирует только черновики.
5. ЛК/площадки: публикация текста через API только после подтверждения.
6. Интерфейс бота: `Текст готов -> Было/станет -> Применить / Поправить`.
7. Защита: фильтр запрещенных слов, обещаний, персональных данных, накрутки ключей.
8. Данные: сохраняет версии черновиков и результат после публикации.
9. Связи: использует каталог, влияет на аналитику и обучение.
10. Эскалация: спорные категории, сертификация, медицина/БАД/детские товары -> человеку.

## 4. Штрихкоды и поставки

1. Делает: готовит черновик поставки, считает количество, создает PDF/этикетки как план до подтверждения владельца.
2. Источники ответа: остатки, прогноз, правила склада, коэффициенты приемки, маркировка.
3. БД: `supplies`, `supply_items`, `barcodes`, `stocks`.
4. Серверы: воркер мониторит окна приемки и коэффициенты, API отдает PDF.
5. ЛК/площадки: реальное создание поставки только через official API и только после `ОК`.
6. Интерфейс бота: `Поставка готова -> Печатать этикетки / Отложить`.
7. Защита: проверка маркировки, количества, даты, склада, лимитов API.
8. Данные: хранит статус черновика, кто подтвердил, что было отправлено.
9. Связи: берет прогноз из модуля 10 и обновляет каталог/остатки.
10. Эскалация: “Честный знак”, сертификаты, спорные товары, платная приемка выше порога -> человеку.

## 5. Отзывы и вопросы покупателей

1. Делает: готовит ответы на отзывы/вопросы, отправляет позитив по кнопке, негатив поднимает человеку.
2. Источники ответа: текст отзыва, карточка товара, история ответов, тон бренда, ограничения площадки.
3. БД: `reviews`, `questions`, `answer_templates`, `feedback`.
4. Серверы: воркер опрашивает новые отзывы, FastAPI подтверждает отправку, Redis соблюдает лимиты.
5. ЛК/площадки: WB/Ozon/ЯМ feedback APIs; без токена или нужной подписки показывает блокер и не подставляет фиктивный отзыв.
6. Интерфейс бота: `Отзыв -> готовый ответ -> Отправить / Поправить / Передать человеку`.
7. Защита: без раскрытия персональных данных, без невыполнимых обещаний и агрессивных ответов.
8. Данные: хранит тон, версию промпта, статус ответа, оценку пользователя.
9. Связи: жалобы кормят аналитику качества, радар рисков и обучение.
10. Эскалация: рейтинг низкий, спор, возврат, угроза суда, мат, крупная сумма -> человеку.

## 6. Разведка конкурентов

1. Делает: следит за публичными ценами/акциями конкурентов и показывает разрыв.
2. Источники ответа: публичные витрины, пользовательские ссылки, внешняя аналитика, без чужих ЛК.
3. БД: `competitors`, `competitor_prices`, `products`.
4. Серверы: аккуратный сбор по расписанию, backoff, robots/лимиты, retry.
5. ЛК/площадки: API при наличии, публичный сбор как вторичный и хрупкий источник.
6. Интерфейс бота: `Конкурент снизил цену -> Ответить?`.
7. Защита: никаких атак, накруток, обходов авторизации, агрессивного парсинга.
8. Данные: хранит источник, время, уверенность, историю цены.
9. Связи: кормит репрайс, SEO и утренний список.
10. Эскалация: данных нет, витрина изменилась, блокировка или низкая уверенность -> честно “нет данных”.

## 7. Репрайс

1. Делает: предлагает новую цену по марже, остаткам, конкурентам и правилам пользователя.
2. Источники ответа: юнитка, конкурентные цены, min/max границы, история продаж.
3. БД: `price_rules`, `price_history`, `products`, `audit_log`.
4. Серверы: пересчет по событию изменения тарифа/остатка/конкурента.
5. ЛК/площадки: изменение цены через API только после `ОК`.
6. Интерфейс бота: `Было X -> станет Y, маржа Z -> Применить`.
7. Защита: нельзя ниже себестоимости+минимальная маржа; анти-демпинг; идемпотентность.
8. Данные: хранит причину рекомендации и эффект после изменения.
9. Связи: модуль 2, 6, 10, 12.
10. Эскалация: большая просадка цены, акция площадки, спорная категория -> человеку.

## 8. Финмонитор

1. Делает: сверяет начисления, удержания, комиссии и выплаты.
2. Источники ответа: финотчеты площадок, ожидаемые комиссии, правила оспаривания.
3. БД: `finance_ops`, `payouts`, `disputes`, `audit_log`.
4. Серверы: ежедневная выгрузка, сверка расхождений, дедлайны споров.
5. ЛК/площадки: official reports; импорт CSV/XLS как безопасный fallback.
6. Интерфейс бота: `Похоже лишнее удержание -> Подготовить спор?`.
7. Защита: доступ только owner/accountant; деньги не выводит и не платит.
8. Данные: хранит доказательства, файл-источник, дедлайн, статус спора.
9. Связи: юнитка, утренние дела, радар тарифов.
10. Эскалация: спор и любые действия с деньгами всегда человеку.

## 9. Антифрод возвратов и память

1. Делает: подсвечивает риск подмены/повреждения при возврате.
2. Источники ответа: факты возвратов, история клиента/товара, правила ПВЗ, пользовательские отметки.
3. БД: `returns`, `client_risk`, `embeddings`, `feedback`.
4. Серверы: мгновенная проверка по событию возврата, память через pgvector.
5. ЛК/площадки: события возврата из API или ручной ввод оператором ПВЗ.
6. Интерфейс бота: `Проверь при клиенте: возможна подмена`.
7. Защита: без дискриминации, только факты; не хранить лишние персональные данные.
8. Данные: хранит признаки риска и объяснение простыми словами.
9. Связи: ПВЗ, обучение, утренние риски.
10. Эскалация: отказ в возврате принимает человек, бот только предупреждает.

## 10. Прогноз спроса и пополнение

1. Делает: прогнозирует спрос, считает точку заказа и готовит черновик пополнения.
2. Источники ответа: продажи, остатки, сезонность, сроки поставки, акции, возвраты.
3. БД: `sales_history`, `forecasts`, `reorder_points`, `supplies`.
4. Серверы: ночной пересчет, отдельный расчет для SKU с достаточной историей.
5. ЛК/площадки: берет продажи и остатки из official API или ручного импорта владельца.
6. Интерфейс бота: `Остатка на 12 дней -> заказать 80 шт?`.
7. Защита: показывает доверие прогноза; не обещает точность.
8. Данные: хранит факт против прогноза, выбранную модель и ошибку.
9. Связи: поставки, репрайс, утренние дела.
10. Эскалация: мало данных, резкая сезонность, новый товар -> человеку.

## 11. ПВЗ: смены, ЗП, маршруты, штрафы

1. Делает: строит график 2/2, считает ЗП за смены, подсвечивает риски штрафов.
2. Источники ответа: список сотрудников, ставки, точки, график, тарифы/штрафы.
3. БД: `pvz_points`, `employees`, `shifts`, `payroll`, `fines`.
4. Серверы: пересчет графика при изменении состава, дневной расчет ЗП.
5. ЛК/площадки: тарифы и начисления из API/отчетов; видео/кассу не заменяет.
6. Интерфейс бота: `График готов -> Утвердить`, `ЗП посчитана -> Показать`.
7. Защита: персональные данные сотрудников только owner/manager; увольнения и выплаты не автоматизируются.
8. Данные: хранит ставку, смену, причину штрафа, кто утвердил.
9. Связи: антифрод, финмонитор, радар правил.
10. Эскалация: трудовой спор, увольнение, выплата, конфликт смен -> человеку.

## 12. Радар правил, оферт и тарифов

1. Делает: отслеживает изменения правил/оферт/тарифов и переводит их в последствия.
2. Источники ответа: официальные страницы, API уведомления, changelog, сохраненная версия.
3. БД: `rule_changes`, `rule_versions`, `tasks`, `audit_log`.
4. Серверы: changedetection/diff worker, RAG summaries, пересчет зависимых модулей.
5. ЛК/площадки: официальные источники приоритетнее чатов и форумов.
6. Интерфейс бота: `Тариф изменился -> маржа упадет на X -> Пересчитать`.
7. Защита: новое не применяется автоматически; только предложение.
8. Данные: хранит diff, источник, дату, уверенность, affected tenants.
9. Связи: юнитка, репрайс, финмонитор, ПВЗ.
10. Эскалация: юридически спорное изменение -> человеку.

## 13. Утренний список дел и оркестратор

1. Делает: каждое утро собирает 3-5 главных дел с готовыми действиями.
2. Источники ответа: сигналы всех модулей, деньги/срочность/риск/история реакций.
3. БД: `tasks`, `task_runs`, `audit_log`, `feedback`.
4. Серверы: Celery beat по timezone tenant, Redis очередь, idempotency key.
5. ЛК/площадки: не ходит напрямую, берет данные через модули.
6. Интерфейс бота: `Дела` и `Сделать все`, но опасное все равно по одному `ОК`.
7. Защита: DO-CONFIRM для цены, денег, поставок, людей, споров.
8. Данные: хранит почему дело попало в топ и что сделал пользователь.
9. Связи: центр продукта.
10. Эскалация: повторная ошибка модуля, сомнение модели, конфликт данных -> человеку/разработчику.

## 14. Касса и учет

1. Делает: готовит сводки и черновики интеграций с кассой/учетом, не заменяет 54-ФЗ/1С.
2. Источники ответа: продажи/возвраты/ПВЗ операции, официальные кассовые провайдеры.
3. БД: `cash_ops`, `receipts`, `reconciliations`, `audit_log`.
4. Серверы: экспорт событий, вебхуки провайдеров, сверка статусов.
5. ЛК/площадки: берет операции из модулей 1/11.
6. Интерфейс бота: `Сводка готова -> Скачать / Передать бухгалтеру`.
7. Защита: чеки, налоги, касса - только интеграция с официальными сервисами.
8. Данные: хранит статус, провайдера, внешний id, ошибку.
9. Связи: ПВЗ, финмонитор, отчеты.
10. Эскалация: фискальная ошибка или налоговый вопрос -> человеку.

## 15. Оплата и студия модулей

1. Делает: считает использование модулей и готовит оплату “плати за нужное”; студия собирает черновик модуля.
2. Источники ответа: usage events, тарифы, подтвержденные модули.
3. БД: `token_usage`, `studio_specs`, `studio_builds`, `audit_log`.
4. Серверы: metering worker, billing webhook, sandbox build runner.
5. ЛК/площадки: не зависит от marketplace API.
6. Интерфейс бота: `Использовано 3 модуля -> Оплатить`.
7. Защита: новый код только в песочнице, без прод-выката без ревью.
8. Данные: хранит usage без лишней персоналки.
9. Связи: все модули пишут usage.
10. Эскалация: списание денег, спор оплаты, новый модуль в прод -> человеку.

## 16. ИИ-надзиратель

1. Делает: контролирует бюджет LLM, качество ответов, prompt injection и здоровье сервисов.
2. Источники ответа: LLM logs без секретов, audit, healthchecks, ошибки модулей.
3. БД: `incidents`, `alerts`, `health_checks`, `token_usage`, `audit_log`.
4. Серверы: middleware, LLM router, health worker, Docker healthcheck.
5. ЛК/площадки: следит за лимитами API, не меняет данные площадок.
6. Интерфейс бота: пользователь видит только простое “сейчас не могу, попробую позже”.
7. Защита: секреты редактируются, injection не становится инструкцией, модель не решает права.
8. Данные: хранит model id, prompt version, cost, статус, без полного секрета.
9. Связи: все LLM и опасные действия.
10. Эскалация: перерасход, повторная галлюцинация, сбой API -> stop/fallback/разработчик.

## 17. Самообучение на оценках

1. Делает: собирает положительные и отрицательные оценки и правки, улучшает тон, приоритеты и шаблоны.
2. Источники ответа: feedback, action result, повторяемость ошибок.
3. БД: `feedback`, `eval_runs`, `semantic_memory`, `audit_log`.
4. Серверы: nightly summarizer, prompt versioning, no auto-deploy.
5. ЛК/площадки: не меняет кабинеты напрямую.
6. Интерфейс бота: после действия `Оценить` и `Что поправить?`.
7. Защита: пользовательская правка не становится системной инструкцией без фильтра.
8. Данные: хранит scoped memory per tenant.
9. Связи: отзывы, SEO, утренний список, антифрод.
10. Эскалация: массовый негатив по модулю -> отключить автоматизацию и открыть баг.

## 18. Авто-радар новинок

1. Делает: следит за новыми API, законами, статьями, репозиториями и предлагает обновить базу знаний.
2. Источники ответа: official changelog, GitHub, Habr, RSS, Telegram channels как вторичный сигнал.
3. БД: `source_changes`, `knowledge_proposals`, `rule_changes`, `audit_log`.
4. Серверы: crawler worker, source ranker, summarizer, approval queue.
5. ЛК/площадки: не применяет изменения к рабочим кабинетам.
6. Интерфейс бота: `3 изменения за неделю -> Применить в знания?`.
7. Защита: непроверенный контент только как черновик; prompt injection фильтруется.
8. Данные: хранит источник, дату, diff, уровень доверия.
9. Связи: радар правил, банк репозиториев, студия модулей.
10. Эскалация: источник неофициальный, противоречие или риск закона -> человеку.

## 19. Реклама и автобиддер

1. Делает: готовит изменение рекламной ставки, стоп перерасхода и дневной план рекламы.
2. Источники ответа: официальные рекламные API WB/Ozon, статистика кампаний, ДРР, маржа и остатки.
3. БД: `ad_campaigns`, `ad_keywords`, `bid_rules`, `bid_history`, `competitor_bids`, `keyword_clusters`.
4. Серверы: воркер собирает статистику, считает ставку, ставит задачу в утренний список.
5. ЛК/площадки: живые изменения ставок только через проверенные официальные методы и после ОК.
6. Интерфейс бота: `Ставка съедает бюджет -> Проверить ставку -> ОК`.
